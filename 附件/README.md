# 附件目录说明

`附件/` 用于存放产品设计过程中引用的原始资料、项目材料、需求参考截图和通用图片资源。

## 目录结构

```text
附件/
├── 01-产品资料/
│   └── {产品线}/
│       └── {文档类型}/
│           └── {资料名称}
├── 02-项目资料/
│   └── {项目名称}/
│       └── {资料类型}/
│           └── {资料名称}/
├── 03-需求参考/
│   └── {产品线}/
│       └── {项目或专题}/
│           └── {需求名称}/
├── 05-通用资源/
│   └── {资源类型}/
└── 99-待整理/
    ├── 历史遗留/
    ├── 外部资料/
    ├── generated_imgs/
    └── 收件箱/
```

## 各目录用途

- `01-产品资料/`：存放监控中心、观测中心等产品线的用户手册、交底书等长期背景资料；
- `02-项目资料/`：存放按客户或项目归档的资料，如项目方案、交付材料、调研截图；
- `03-需求参考/`：存放具体需求在设计过程中使用的截图、对标图、原始参考图；
- `05-通用资源/`：存放可跨需求复用的公共素材，如 logo、架构图、品牌图；
- `99-待整理/`：存放暂未确认归属的资料，后续再归档到正式目录；
- `99-待整理/收件箱/`：Obsidian 新粘贴截图的默认落点，避免截图直接散落到仓库根目录。

补充说明：
- 已废弃的系统机制文档、历史系统资料可放入 `99-待整理/历史遗留/系统资料/`，不再作为当前知识库正式入口。

## 当前归档规则

1. 用户手册、交底书等正式背景文档，优先放入 `01-产品资料/`，并按 `监控中心/观测中心 → 用户手册/交底书` 归档；
2. 与客户项目强相关的资料，优先放入 `02-项目资料/`；
3. 某个需求说明文档直接引用的截图，统一放入 `03-需求参考/`；
4. 临时导出的图片、历史零散截图，先放入 `99-待整理/`，确认用途后再迁移；
5. 同一份资料的 Markdown、原图 `media/`、补充文件尽量放在同一个资料目录下，不要拆散。

## 截图整理流程

1. 在 Obsidian 中粘贴外部截图后，文件会先进入 `附件/99-待整理/收件箱/`；
2. 执行以下命令自动整理：

```bash
python3 scripts/正式工具/整理附件.py
```

3. 脚本会优先按“被哪个 Markdown 引用”自动归档：
   - 需求文档图片 → `附件/03-需求参考/...`
   - `附件/` 内资料图片 → 对应资料目录下的 `media/`
   - 复盘图片 → `附件/04-复盘配图/...`
   - 无法判断归属的图片 → 保留在 `附件/99-待整理/收件箱/` 或转入人工处理区

如需先预览结果，可执行：

```bash
python3 scripts/正式工具/整理附件.py --dry-run
```

## Office 转 Markdown

当 `01-产品资料/` 下新增 Word 或 Excel 文件后，可执行以下命令批量转换：

```bash
python3 scripts/正式工具/转换产品资料.py
```

仅预览将要生成的文件路径：

```bash
python3 scripts/正式工具/转换产品资料.py --dry-run
```

转换规则：

- `.docx` → 在原目录下生成同名文件夹，输出 `同名.md` 与 `media/`
- `.xlsx` / `.xls` → 在原目录下生成同名 `.md`

## 维护建议

- 新增资料时，优先按“产品线 / 项目 / 需求”判断归属，不确定时先放 `99-待整理/`；
- 需要给需求说明文档配图时，优先复用 `03-需求参考/` 里的现有截图，避免根目录散落新文件；
- 后续如需建设可复用知识库，可在 `product-context/` 中新增对应的背景摘要文件，而不是直接把 `附件/` 当作知识库使用。
