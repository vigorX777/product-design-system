# 设计原则库

> 从实际需求案例中提炼的可复用设计原则，每条原则必须有实际案例支撑。

## 字段说明

| 字段 | 说明 |
|------|------|
| **编号** | DP-001 格式，递增 |
| **名称** | 简洁的原则名称（4-10 字） |
| **描述** | 1-3 句话说明原则内容 |
| **来源案例** | 引用的需求路径（必填） |
| **适用范围** | 适用的场景或组件类型 |
| **反例** | 违反此原则的典型做法 |

## DP-001: 操作前置条件判断

- **描述**: 为了避免用户进入操作流程后才发现无法继续，应当在点击操作按钮时先判断前置条件（如是否已有关联任务），根据判断结果引导至不同的操作分支。
- **来源案例**: `v4.5.x/监控策略目标显示无采集任务关联信息/`
- **适用范围**: 所有涉及操作前置条件判断的场景，特别是创建/编辑等复杂表单操作前。
- **反例**: 用户点击"配置采集任务"直接进入新建页面，填完一堆信息后才提示"该监控项已有采集任务，请去编辑"。

## DP-002: 状态异常前置展示

- **描述**: 为了提高异常发现效率，应当将"无采集任务"等异常状态在列表或详情的默认视图中直接可见（如通过 tooltip 或状态标签），不需要用户点击进入深层页面才发现。
- **来源案例**: `v4.5.x/监控策略目标显示无采集任务关联信息/`
- **适用范围**: 监控目标列表、策略详情列表等需要展示对象健康或配置状态的场景。
- **反例**: 列表中只显示"异常"状态，用户必须点击"查看详情"进入下一级页面才能看到是因为"未配置采集任务"导致的异常。

## DP-003: 新增数据行高亮

- **描述**: 为了让用户明确感知到数据添加或关联成功，应当将新增加的数据行置顶显示，并使用浅色背景（如浅蓝色）进行高亮，以区分已有数据。
- **来源案例**: `v4.5.x/监控策略目标显示无采集任务关联信息/`
- **适用范围**: 列表页面的增量添加、从其他页面带入数据并自动选中的场景。
- **反例**: 新增的数据混在原有列表中，没有任何视觉提示，用户需要自己去翻找确认是否添加成功。

## DP-004: 模板复用优先于从头编写

- **描述**: 开始新需求的原型设计时，应首先检查 `product-context/组件模板/` 是否有可直接使用的基础模板，以模板为起点修改，而不是从空白 HTML 文件开始编写。可节省约 80% 的基础样式与组件编写时间。
- **来源案例**: `product-context/需求复盘/验证测试-复盘.md`（端到端验证，2026-02-26）
- **适用范围**: 所有需要原型设计的新需求。优先选择最接近目标页面结构的模板（表格页 / 弹窗 / 侧滑）。
- **反例**: 从空白 HTML 开始编写，重复定义 CSS 变量、基础样式和通用组件，导致每次原型风格略有差异。

## DP-005: 配置依赖跨页面引导

- **描述**: 当某页面功能依赖另一模块的前置配置时，不能仅展示「未配置」或「不可用」状态，必须在该页面直接提供跳转入口（引导横条 + 文字按钮），让用户无需自行寻找配置路径即可完成前置步骤。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`
- **适用范围**: 任何功能存在跨页面配置依赖的场景，尤其是新接入的功能模块对已有模块的配置存在依赖时。
- **反例**: 拓扑分层管理页直接显示「关联关系字段：未配置」，用户不知道去哪里配置，需要自己摸索找到「对象模型管理」页面。

## DP-006: 锁定项与可选项视觉区分

- **描述**: 在多选场景中，系统内置/强制关联的锁定项与用户自由增删的可选项必须在视觉上可区分。锁定项的 tag 无删除按钮，下拉选项置灰（`is-disabled`），可选项 tag 带「×」且下拉可自由勾选。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`
- **适用范围**: 所有多选组件中存在「强制选中项」的场景，如系统内置采集插件、默认关联模型等。
- **反例**: 锁定项与可选项外观完全相同，用户点击「×」后才发现无法删除（或意外删除了不应删除的项），体验困惑。

## DP-007: 异步操作行级状态反馈

- **描述**: 耗时操作（如删除、启停）不应只有全局 loading，应在操作行上直接展示进度状态——行背景变灰、文字变淡、按钮文案改为「操作中...」、相关操作禁用——让用户明确感知「哪一行正在处理什么」。
- **来源案例**: `v4.5.x/监控中心/采集任务删除效果优化/`
- **适用范围**: 所有列表页的异步行级操作（删除、启停、提交等），尤其是操作耗时超过 300ms 或需要后端异步处理的场景。
- **反例**: 触发删除后只弹全屏 loading 或无任何状态变化，用户无法判断哪行正在处理，容易重复点击或误判操作结果。

## DP-008: 失败可重试，原地恢复上下文

- **描述**: 异步操作失败时，行应完整恢复到操作前的状态（样式、开关状态、按钮文案），并在行内显示失败标识（红色感叹号图标）+ Tooltip 说明失败原因，用户无需刷新或重新勾选即可直接重试。
- **来源案例**: `v4.5.x/监控中心/采集任务删除效果优化/`
- **适用范围**: 所有存在失败可能的异步行级操作；开关类操作需记录操作前状态（`previousToggleState`）用于恢复。
- **反例**: 操作失败后只弹全局 Toast，行保持灰色或置灰状态，用户需要刷新页面才能重试，操作前的勾选状态也已丢失。

## DP-010: 拓扑聚合最细粒度优先

- **描述**: 拓扑图原型中，集群聚合应从「每个集群独立折叠/展开」的最细粒度出发设计，而不是从整层统一聚合开始再拆分。整层聚合在原型验证阶段看似简单，但一旦需要区分单个集群的告警或状态，重构成本极高（涉及 HTML 结构、CSS、JS 连线逻辑全面改造）。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`（CHANGELOG #28→#30，整层聚合→独立聚合组重构）
- **适用范围**: 全景拓扑、依赖关系图等任何需要折叠/展开节点组的原型设计。
- **反例**: 先把一整层（如服务层全部 9 个集群）用一个折叠/展开按钮控制，后来发现用户需要精确感知"哪个集群有告警"，不得不推倒重构整层的 HTML/CSS/JS。

## DP-011: 信息层次随操作状态递进

- **描述**: 节点卡片、角标等信息标注的密度应与用户当前操作意图对应——折叠态用于「快速总览」（只看规模/告警总数），展开态用于「诊断定位」（看具体实例和明细）。同一字段在不同状态下可以选择性显示，不必强求折叠/展开态信息完全一致。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`（Badge 格式：折叠态显示总数 N，展开态 footer 显示「告警数/总数」X/N；悬浮卡片：折叠态聚合卡片不显示实例信息字段，仅展开态和实例卡片显示）
- **适用范围**: 所有具有折叠/展开两态的列表、节点、面板组件。
- **反例**: 折叠态和展开态卡片显示完全相同的字段，折叠态卡片塞入过多细节字段，导致「总览」功能失效，用户反而要逐一展开才能快速扫描状态。

## DP-012: 跨需求共用模块独立拆出

- **描述**: 当某个功能模块（如配置管理页面）被两个及以上需求共同依赖时，应将其拆分为独立需求文件夹，而不是在其中一个需求中维护后被另一个引用。独立后有独立的 CHANGELOG，变更溯源更清晰，两个依赖方可分别跟进进度，避免因"放在哪个需求里"而产生认知混乱。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求3：拓扑分层管理/`（从需求1 和需求2 中各自含有的「拓扑分层管理」章节拆出，合并为独立需求文件夹）
- **适用范围**: 任何被多个需求引用或依赖的配置模块、公共页面或共用功能点。
- **反例**: 「拓扑分层管理」放在需求1 文档里，需求2 通过链接引用需求1，后来需求1 和需求2 各自迭代时，分层管理的变更夹杂在需求1 的 CHANGELOG 里，无法独立追溯。

## DP-009: 局部锁定，不阻断全局导航

- **描述**: 异步操作进行中应仅锁定与数据变更直接相关的入口（增删改按钮、复选框、相关开关），页面其余导航（侧边栏、顶部菜单、搜索框、分页）保持可用，避免用户感到被「困住」无法切换上下文。
- **来源案例**: `v4.5.x/监控中心/采集任务操作逻辑优化/`
- **适用范围**: 所有需要等待后端处理完成才能继续操作的异步流程，包括批量操作、启停任务等。
- **反例**: 触发操作后整页遮罩或全部按钮禁用，用户无法切换侧边栏菜单或搜索其他内容，操作感受被强制阻断。

## DP-013: ⚠️ 原位标注 + 末尾汇总双轨制

- **描述**: 文档中的待定事项在原位用 ⚠️ 标注保留上下文，同时在文档末尾「遗留项」章节按类别汇总索引并标注所在章节位置。评审时既能逐条阅读上下文、又能快速获取全局视图。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`（遗留项重写，将 10 处 ⚠️ 按 APM 建模/听云指标/听云跳转三类汇总）
- **适用范围**: 所有含待定设计点（⚠️）的需求说明文档，尤其待定事项超过 3 处时分类汇总价值更高。
- **反例**: 只在文档末尾的遗留项中罗列待定事项编号，阅读时需反复跳转才能理解每个待定事项的具体上下文；或只在原文中标注 ⚠️，没有汇总视图，评审时容易遗漏。

## DP-014: 遗留项三段式结构

- **描述**: 遗留项章节统一分为「待定事项」「前提操作」「注意事项」三个小节——待定事项记录尚未确认的设计点（对应 ⚠️），前提操作记录部署/配置阶段的必要步骤（可作为交付检查清单），注意事项记录不影响功能但需关注的约束。即使文档中没有 ⚠️ 待定事项，遗留项章节仍应存在，用于记录前提操作和注意事项。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求1：应用、服务层支持通过CMDB映射第三方系统的实例【中广核】/`（10 处 ⚠️ + 5 项前提操作 + 2 条注意事项）；`v4.5.x/观测中心/中广核项目需求/需求3：拓扑分层管理/`（无 ⚠️ 但有 3 项前提操作 + 2 条注意事项，作为"无待定项也写遗留项"的案例）
- **适用范围**: 所有需求说明文档的遗留项章节。
- **反例**: 遗留项只写"待确认事项"，部署前提（如升级脚本、手动关联步骤）散落在正文各处或口头传达，交付时遗漏配置步骤。

## DP-015: 跨需求依赖显式声明

- **描述**: 当一个需求的实现依赖另一个需求的完成时，应在遗留项的「前提操作」中显式声明依赖关系（如"依赖需求 X 完成"），而不是仅在背景描述中提及。显式声明便于项目管理排期和交付检查。
- **来源案例**: `v4.5.x/观测中心/中广核项目需求/需求2：新增一层RUM 终端层，支持通过CMDB映射第三方系统的实例【中广核】/`（遗留项第一条前提操作：「第一期需求（应用、服务层 CMDB 映射）完成」）
- **适用范围**: 所有存在跨需求依赖关系的需求文档，尤其是分期交付的项目。
- **反例**: 终端层需求文档只在背景部分写"本需求基于第一期能力扩展"，没有在遗留项中显式列出依赖，项目排期时遗漏前置条件，导致终端层提前进入开发但第一期尚未完成。
