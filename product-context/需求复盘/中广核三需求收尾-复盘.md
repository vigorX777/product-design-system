## 关键阻塞与解法

### 1. 整层聚合 → 独立聚合组的重构成本
**问题**：最初原型将服务层所有集群作为一个整体聚合/展开，后来确认需要每个集群独立折叠/展开，涉及 HTML/CSS/JS 全面重构（CHANGELOG #30）。  
**解法**：按「cluster-group」单元重构，每个集群的 HTML 结构自包含（菱形节点 + 实例行），连线动态绘制绑定到 cluster-group 状态，最终服务层 9 组、中间件 4 组、数据库 7 组全部独立运行。  
**启示**：拓扑图原型建议第一版就按「最细粒度可控」原则设计，从独立组出发比从整层出发的后续改造成本低。

### 2. 截图中 hover 卡片无法通过 Playwright 原生触发
**问题**：Playwright 的 `hover()` 动作对部分节点（如应用层节点 `#app-cmdb-1`）无法触发悬浮卡片，因为 `matchHoverTarget` 只绑定了 4 种节点类型，应用层节点未被覆盖。  
**解法**：通过 `browser_evaluate` 手动注入卡片 HTML 到 `#node-hover-card`，添加 `.is-visible` 类并用 `getBoundingClientRect()` 动态定位，再截图。  
**启示**：截图脚本需要了解原型的 hover 绑定机制，不能假设所有节点都能通过 hover 触发卡片。建议在原型说明或 README 中注明哪些节点类型支持 hover 卡片。

### 3. 集群聚合规则抽屉 overflow 冲突
**问题**：集群聚合规则抽屉集成到观测拓扑原型后，CSS 前缀不隔离会与主页样式冲突（CHANGELOG #37）。  
**解法**：所有抽屉相关 CSS 统一使用 `.agg-*` 前缀命名空间，独立于主页 CSS，互不影响。  
**启示**：将独立功能模块（如抽屉、弹窗、浮层）集成到综合原型时，必须使用 CSS 命名空间隔离，不能共用通用类名。

## 提炼的模式 / 原则（候选）

> 以下为候选条目，经人工确认后追加到对应知识文件。

**设计原则候选**:
- [ ] **拓扑聚合最细粒度优先**（DP-010）: 拓扑图原型中，集群聚合应从「每个集群独立折叠/展开」的最细粒度出发设计，而不是从整层统一聚合开始再拆分。整层聚合在原型验证阶段看似简单，但一旦需要区分单个集群的告警或状态，重构成本极高。（来源：需求1 CHANGELOG #28→#30）
- [ ] **信息层次随操作状态递进**（DP-011）: 节点卡片/标注的信息密度应与用户当前操作意图对应——折叠态用于「快速总览」（只看规模/告警总数），展开态用于「诊断定位」（看具体实例和明细）。同一字段在不同状态下可以隐藏或展示，不必强求一致。（来源：需求1 Badge 格式决策 + 悬浮卡片折叠/展开策略）
- [ ] **跨需求共用模块独立拆出**（DP-012）: 当某个功能模块被两个及以上需求依赖时，应拆分为独立需求文件夹，而不是在其中一个需求中维护后被另一个引用。独立后有独立的 CHANGELOG，变更溯源更清晰，也便于不同需求单独跟进进度。（来源：需求3「拓扑分层管理」从需求1/2 拆出）

**交互模式候选**:
- [ ] **集群菱形双态节点**（IP-009）: 集群聚合节点采用菱形样式，折叠态显示聚合总数 badge，双击展开后上半行显示集群菱形小节点（代表集群本身）、下半行显示普通实例节点，连线动态切换。鼠标悬浮对折叠/展开态显示不同信息层次的卡片。（来源：需求1 拓扑视图展示）
- [ ] **嵌入式侧滑抽屉集成**（IP-010）: 将独立功能的抽屉面板集成到综合原型中——通过工具栏图标（如「⚙」）触发侧滑打开，所有抽屉 CSS 使用独立前缀命名空间（`.agg-*`）与主页隔离，确保两者样式不相互污染。（来源：需求1 集群聚合规则抽屉集成）

**原型技术候选**:
- [ ] Playwright 截图时，hover 卡片可通过 `browser_evaluate` 手动注入 HTML + `.is-visible` 类 + `getBoundingClientRect()` 定位来触发，无需依赖原生 mouseover 事件绑定。
